# Architecture Decision: Arcium CPI Strategy

## Date: 2026-03-11 (Updated 2026-03-12)

## Problem

The `#[arcium_program]` macro (which auto-generates `QueueCompAccs` trait implementations, callback type generation, and `queue_computation` CPI helpers) **conflicts with the `session-keys` crate's `#[derive(Session)]` macro**. Both macros modify/parse the program module structure, and they are incompatible.

Additionally, even the standalone attribute macros (`#[queue_computation_accounts]`, `#[init_computation_definition_accounts]`, `#[callback_accounts]`) depend on `ID`/`ID_CONST` being in scope from the `#[arcium_program]` module annotation. The PDA derivation macros (`derive_mxe_pda!()`, `derive_mempool_pda!()`, etc.) have the same dependency.

## Options Explored

### Option A: Use `#[arcium_program]` macro
- Requires removing session-keys dependency → breaks 1-click gameplay
- **Not viable**

### Option B: Use standalone attribute macros without `#[arcium_program]`
- Attempted: `#[queue_computation_accounts("shuffle_and_deal", payer)]` on struct
- **Failed:** Macro generates `__client_accounts_*` structs expecting crate-root symbols from `#[arcium_program]`
- Also tried `#[init_computation_definition_accounts]` — same issue
- `derive_mxe_pda!()` uses bare `ID` (not `crate::ID`) — requires `#[arcium_program]` scope
- **Not viable without `#[arcium_program]`**

### Option C: Manual trait implementations (CHOSEN)
- Manually implement `QueueCompAccs` and `InitCompDefAccs` traits
- Use `arcium_client::pda::*` functions instead of PDA macros
- Import types from `arcium_client::idl::arcium::*` (generated by `declare_program!`)
- Define `ArciumSignerAccount` locally (simple 9-byte account with bump)

## Decision: Option C — Manual Trait Implementations

### What We Had to Build Manually

**1. `ArciumSignerAccount` struct** — normally generated by `#[arcium_program]`:
```rust
#[account]
pub struct ArciumSignerAccount {
    pub bump: u8,
}
```
Seeds: `[b"ArciumSignerAccount"]` under OUR program ID. Space = 9 bytes (8 discriminator + 1 bump).

**2. `QueueCompAccs` trait** on `ArciumDeal` struct:
```rust
impl<'info> QueueCompAccs<'info> for ArciumDeal<'info> {
    fn comp_def_offset(&self) -> u32 { COMP_DEF_OFFSET_SHUFFLE }
    fn queue_comp_accs(&self) -> QueueComputation<'info> { /* map accounts */ }
    fn arcium_program(&self) -> AccountInfo<'info> { ... }
    fn mxe_program(&self) -> Pubkey { PROGRAM_ID }
    fn signer_pda_bump(&self) -> u8 { self.sign_pda_account.bump }
}
```

**3. `InitCompDefAccs` trait** on `InitShuffleCompDef`, `InitRevealCompDef`, `InitShowdownCompDef`:
- Returns `Vec<Parameter>` and `Vec<Output>` matching the `.idarc` circuit interface
- Uses `arcium_client::idl::arcium::types::{Parameter, Output}` enums directly

**4. PDA derivation** — using `arcium_client::pda::*` instead of macros:
```rust
arcium_client::pda::mxe_acc(&PROGRAM_ID)                           // MXE account
arcium_client::pda::computation_definition_acc(&PROGRAM_ID, offset) // comp def
```

### On-Chain Instructions (Rust/Anchor)
1. **`init_shuffle_comp_def`** / **`init_reveal_comp_def`** / **`init_showdown_comp_def`** — register circuits with Arcium (one-time setup)
2. **`arcium_deal`** — validates Starting phase, builds ArgBuilder args, CPI to `queue_computation`, transitions to AwaitingDeal
3. **`reveal_community_callback`** — callback from MPC: writes plaintext community cards to Table
4. **`reveal_showdown_callback`** — callback from MPC: writes plaintext hole cards to SeatCards

### TypeScript Crank
The `crankArciumDeal()` in `crank-service.ts` builds the full instruction with all Arcium accounts and sends it as a single TX. The Rust handler does the actual CPI to Arcium's `queue_computation`.

### Why This Works
- Session keys preserved for player actions (check/call/raise/fold)
- Full CPI happens on-chain (more secure than split TS approach)
- Manual trait impl is ~100 lines of boilerplate but well-understood
- `arcium_client::pda::*` functions are stable public API (less fragile than macros)

### Key Imports Pattern
```rust
// From arcium-anchor (CPI functions + constants)
use arcium_anchor::{queue_computation, init_comp_def, ArgBuilder, SIGN_PDA_SEED, ...};
use arcium_anchor::traits::{QueueCompAccs, InitCompDefAccs};

// From arcium-client (types generated by declare_program!)
use arcium_client::idl::arcium::{
    accounts::{MXEAccount, ComputationDefinitionAccount, Cluster, FeePool, ClockAccount},
    cpi::accounts::QueueComputation,
    program::Arcium,
    types::{Parameter, Output, CallbackInstruction},
    ID_CONST as ARCIUM_PROG_ID,
};
use arcium_client::pda as arcium_pda;
```

## Callback Implementation (Updated 2026-03-13)

### HasSize::SIZE — Critical for Deserialization

`SignedComputationOutputs<T>` custom deserialization reads exactly `T::SIZE` bytes as raw output,
then 64 bytes for BLS signature. Anchor rejects any leftover bytes (error 102).

**⚠️ CRITICAL FINDING: Raw MPC output ≠ SharedEncryptedStruct format!**

MPC nodes output **only 32-byte Rescue ciphertexts per encrypted value** — no pubkey, no nonce.
The `SharedEncryptedStruct` format (32 pubkey + 16 nonce + 32 ct = 80 bytes) and
`MXEEncryptedStruct` format (16 nonce + 32 ct = 48 bytes) are **TypeScript client-side only**.
`arcium-macros value_size_in_bytes()` uses these struct sizes → generates SIZE=1328 → **WRONG**.

For `shuffle_and_deal` (17 encrypted outputs):
- **SIZE = 17 × 32 = 544 bytes** (confirmed working)

Verified empirically from Arcium node logs:
- BLS message length = 554 bytes
- BLS message = raw_output + slot(8) + slot_counter(2) (from `get_slot_and_slot_counter_bytes`)
- raw_output = 554 - 10 = 544 = 17 × 32 ✓

### BLS Verification — Localnet Limitation

`verify_output_raw()` fails on localnet with error 6001 (`BLSSignatureVerificationFailed`).
Root cause: cluster BLS key mismatch on localnet Docker nodes.

**Workaround:** Use CPI context validation via `validate_arcium_callback_context()`:
- Checks preceding instruction is Arcium's `callbackComputation` program
- Prevents spoofing — only Arcium program can invoke our callback
- TODO: Re-enable BLS verification for devnet/mainnet

### Raw Output Parsing

Raw output = N × 32-byte ciphertexts in uniform layout. Each output at `idx * 32`.
No pubkey/nonce metadata in raw output — nonces derived client-side from computation parameters.

### Key Source Files (arcium-anchor 0.8.5)
- `arcium-anchor/src/lib.rs` — `SignedComputationOutputs` enum, `get_slot_and_slot_counter_bytes`
